## Status

**Completed** (2025-12-17)

## Story

**As a** framework user,  
**I want** project-type templates (archetypes) that define sane defaults,  
**so that** initialization produces consistent, best-practice configuration for the kind of project I’m building.

## Acceptance Criteria

1. `templates/project_types/` directory exists
2. A project-type template format is defined (YAML) and documented
3. At least 5 project-type templates are created:
   - api-service
   - web-app
   - cli-tool
   - library
   - microservice
4. Templates include defaults appropriate to type (workflow suggestions, quality thresholds, docs skeleton hints)
5. Templates are valid YAML and compatible with the existing merge system

## Dev Notes

### Existing System Context

- Tech stack templates exist under `templates/tech_stacks/` and are applied during init (see `docs/stories/31.4.init-integration.md`)
- Merge system and variable expansion exist (see `docs/stories/31.3.template-merging-system.md`)

### Integration Approach

- Keep project-type templates orthogonal to tech-stack templates:
  - Project type answers “what kind of system is this?”
  - Tech stack answers “what technologies does it use?”
- Project-type templates should merge cleanly with tech-stack templates and defaults, with user config/customizations last.
