Custom Workflows

Create custom workflows by defining YAML files in your workflows/ directory.

Workflow Structure

Custom Workflow Example 
workflow:
  id: my-custom-workflow
  name: "My Custom Workflow"
  description: "Custom workflow for specific needs"
  version: "1.0.0"
  
  type: brownfield  # or greenfield
  auto_detect: true
  
  settings:
    quality_gates: true
    code_scoring: true
    context_tier_default: 2
  
  steps:
    - id: step1
      agent: planner
      action: create_stories
      context_tier: 1
      creates:
        - stories/
      requires: []
      next: step2
      
    - id: step2
      agent: implementer
      action: write_code
      context_tier: 3
      requires:
        - stories/
      creates:
        - src/
      next: review
      
    - id: review
      agent: reviewer
      action: review_code
      context_tier: 2
      requires:
        - src/
      scoring:
        enabled: true
        thresholds:
          overall_min: 70
      gate:
        condition: "scoring.passed == true"
        on_pass: complete
        on_fail: step2
        
    - id: complete
      agent: orchestrator
      action: finalize
      context_tier: 1

Running Custom Workflows

Run Custom Workflow 
tapps-agents workflow run workflows/my-custom-workflow.yaml

Context Tiers

Context tiers control how much code context is provided to agents, balancing completeness with token efficiency.

Tier 1: Core Context (~500 tokens)

  • File structure
  • Function signatures
  • Class definitions
  • Imports/exports

Used by: Analyst, Planner, Reviewer (initial), Ops

Tier 2: Extended Context (~2000 tokens)

  • Everything in Tier 1
  • Function bodies
  • Related files
  • Test files

Used by: Architect, Designer, Reviewer (detailed), Tester

Tier 3: Full Context (~10000 tokens)

  • Everything in Tier 2
  • Complete file contents
  • Git history
  • Documentation
  • Cross-references

Used by: Implementer, Improver

Configuring Context Tiers

Context Configuration 
# In .tapps-agents/config.yaml
context:
  tiered_enabled: true
  default_tier: "tier1"  # tier1, tier2, or tier3
  cache_ttl: 300  # Cache time-to-live in seconds

Hardware Profiles

Hardware profiles optimize performance for different system configurations, especially important for low-power systems like Intel NUCs.

Available Profiles

  • auto - Auto-detect (default)
  • nuc - Intel NUC / low-power systems
  • development - Development machines
  • workstation - High-performance workstations
  • server - Server environments

Setting Hardware Profile

Hardware Profile Commands 
# Check current profile
tapps-agents hardware-profile

# Set profile
tapps-agents hardware-profile --set nuc

Performance Tuning

Token Optimization

  • Use appropriate context tiers (Tier 1 for most tasks)
  • Enable tiered context caching
  • Break large tasks into smaller workflows
  • All LLM operations handled by Cursor Skills automatically

Caching Strategy

Context caching reduces redundant processing:

Cache Configuration 
# In .tapps-agents/config.yaml
context:
  cache_ttl: 300  # 5 minutes for Tier 1
  # Tier 2: 120 seconds (2 minutes)
  # Tier 3: 60 seconds (1 minute)

Model Selection

All LLM operations are handled by Cursor Skills, which use the developer's configured model in Cursor. No model selection or API keys are required - Cursor handles all model management automatically.

Integration Patterns

CI/CD Integration

Integrate workflows into your CI/CD pipeline:

CI/CD Example 
# .github/workflows/quality-check.yml
- name: Run Quality Workflow
  run: |
    tapps-agents workflow quality
    # Check exit code for quality gate results

Pre-commit Hooks

Pre-commit Hook 
# .git/hooks/pre-commit
#!/bin/bash
tapps-agents reviewer score $(git diff --cached --name-only --diff-filter=ACM | grep '\.py$')
# Exit with error if score below threshold
